Hello. This is a new C++ plugin project using the JUCE library, called Gloubiboulga. My goal is to create a glitch plugin according to the following specifications. 


* Parameters of the plugin :

- RandomSeed (int)

- GlitchDurMean (int in the range [1, currentPositionInfo.timeSigNumerator])

- GlitchDurStdDev (int in the range [0, currentPositionInfo.timeSigNumerator-1])

- GlitchProb (float in [0,1])

- BufferStartPosMean (int in the range [0, currentPositionInfo.timeSigNumerator-1])

- BufferStartOffsetMean (int in the range [0,3])

- BufferStartPosStdDev (int in the range [0, currentPositionInfo.timeSigNumerator-1])

- BufferStartOffsetMean (int in the range [0,3])

- CutProb32nd, CutProb16th, CutProb8th, CutProb4th (float in [0,1])

- CutProbNormal, CutProbDotted, CutProbTripled (float in [0,1])

- CutLengthMean (float in [0,1])

- CutLengthStdDev (float in [0,1])

- CycleDuration (int in the range [1,16])


* Logic :

The plugin is aware of the current tempo from the DAW. It continuously keeps in a buffer the last measure based on currentPositionInfo.bpm, currentPositionInfo.timeSigNumerator, and currentPositionInfo.timeSigDenominator, which are defined in juce namespace. This buffer is continuously used to feed the output buffer following these

At each parameter change, the plugin creates a series of instructions that will determine how the incoming sound will be glitched during the next CycleDuration measures. The random seed is here to get a deterministic and reproductible cycle if the same parameters are used.

Here is how the cycle is made:

- Select randomly a duration, named curStepDuration using a normal gaussian law with mean value determined by GlitchDurMean and standard deviation GlitchDurStdDev. The result has to be an integer in the range [1, currentPositionInfo.timeSigNumerator]. It has to be rounded and cutted to fit in this range. The duration unit is the interval determined by the value of currentPositionInfo.timeSigDenominator.

- Select a random number r between 0 and 1, so that a bool can be calculated:

isGlitching=(r<GlitchProb)

If isGlitching is false, no modification of the audio input buffer is made during the duration of curStepDuration. If true, we will apply glitching.

We repeat this until the total duration of CycleDuration measures is reached (there is a end case to treat: if the selected duration is larger than the remaining, we cut it to the remaining).

Once we have selected how we cut the cycle, and when glitching is applied, we must determine how glitch is applied:

- Select randomly a starting position (a float number) of a portion of the recorded buffer, using BufferStartPosMean and BufferStartPosStdDev. The start position unit is determined by currentPositionInfo.timeSigDenominator.

- Select randomly an offset (float) to which the starting position is modified using BufferStartOffsetMean and BufferStartPosStdDev. The start offset unit is 1/4th the duration determined by currentPositionInfo.timeSigDenominator.

- The actual start position startPos is determined by the combination of the starting position and the offset. It is rounded and clipped to the range [0, currentPositionInfo.timeSigNumerator].

Once the actual start position is choosen, we determine how it is used for the actual plugin output. It will be repeatedly played every repeatDur, which is randomly determined using CutProb32nd, CutProb16th, CutProb8th, CutProb4th, CutProbNormal, CutProbDotted and CutProbTripled. The probabilities CutProb32nd, CutProb16th, CutProb8th, CutProb4th determines which chance one 1/32th, 1/16th, 1/8th and 1/4th has to be selected respectively to the other. Then CutProbNormal, CutProbDotted, CutProbTripled determines if it is a normal duration, a dotted or triplet duration which is selected.

And finally, CutLengthMean and CutLengthStdDev determine the probability distribution of which portion will be effectively played, the remaining part will be silence.

The sound starting at startPos in the buffer will the be repeatedly played every 1/nth subdivision (evetually dotted or triplet), using amplitude enveloppe determined by cutLength.

This cycle of cutting parts is repeated until we have covered all CycleDuration measures.

This is what is effectively used by the audio processing, unitl a parameter is changed and a new cycle is calculated.

This would necessitate to implement a random number generator with configurable random seed, and capable of generating a nomal distribution, we could use the Box-Muller transform for that. What do you think?
